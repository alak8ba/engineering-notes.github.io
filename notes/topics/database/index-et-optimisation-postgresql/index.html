    Index et optimisation des requetes PostgreSQL

    Pourquoi les index ?

    Sans index, PostgreSQL fait un sequential scan : il lit chaque ligne de la table une par une pour trouver ce qui correspond a la condition. Sur 10 millions de lignes, c'est lent.

    Un index est une structure de donnees supplementaire (B-Tree par defaut) qui permet de localiser directement les lignes sans tout lire.

    Sans index : O(n)   -> lit toutes les lignes
Avec index : O(log n) -> suit l'arbre pour trouver directement

    Types d'index PostgreSQL

    B-Tree (defaut)

    Le plus courant. Adapte pour les comparaisons (=, <, >, BETWEEN, LIKE 'abc%').

    CREATE INDEX idx_individu_reference ON T_INDIVIDU(NUM_REFERENCE);
CREATE INDEX idx_individu_statut    ON T_INDIVIDU(STATUT);

    Hash

    Uniquement pour l'egalite (=). Plus rapide que B-Tree sur ce cas precis, mais non persistant apres crash avant PostgreSQL 10.

    CREATE INDEX idx_hash_reference ON T_INDIVIDU USING HASH(NUM_REFERENCE);

    GIN (Generalized Inverted Index)

    Pour les recherches dans des tableaux, JSONB, ou full-text search.

    -- Recherche full-text
CREATE INDEX idx_gin_nom ON T_INDIVIDU USING GIN(to_tsvector('french', NOM_PRENOM));

-- Recherche dans JSONB
CREATE INDEX idx_gin_data ON ma_table USING GIN(data_json);

    BRIN (Block Range INdex)

    Pour les tres grandes tables avec donnees naturellement ordonnees (dates, sequences). Tres petit en taille, adapte pour les colonnes d'audit (created_at, DATE_TRAITEMENT).

    CREATE INDEX idx_brin_traitement ON T_INDIVIDU USING BRIN(DATE_TRAITEMENT);

    Partial Index

    Index sur un sous-ensemble de lignes. Tres efficace si on requete souvent sur une condition specifique.

    -- Index uniquement sur les KO (les plus recherches pour analyse)
CREATE INDEX idx_individu_ko ON T_INDIVIDU(NUM_REFERENCE)
WHERE STATUT = 'KO';

-- Index uniquement sur les non traites
CREATE INDEX idx_commande_pending ON T_COMMANDE(CREATED_AT)
WHERE STATUT = 'PENDING';

    Composite Index (multi-colonnes)

    Un index sur plusieurs colonnes. L'ordre des colonnes est crucial.

    CREATE INDEX idx_individu_statut_date ON T_INDIVIDU(STATUT, DATE_TRAITEMENT);

    Regle du prefixe gauche : un index (A, B, C) est utilise pour : - WHERE A = ? - WHERE A = ? AND B = ? - WHERE A = ? AND B = ? AND C = ?

    Mais PAS pour : - WHERE B = ? (sans A) - WHERE C = ? (sans A et B)

    Quand creer un index ?

    Creer un index sur :

    • Cles primaires (auto-cree par PostgreSQL)
    • Cles etrangeres (PostgreSQL ne les cree PAS automatiquement)
    • Colonnes de filtre frequentes (WHERE statut = ?, WHERE pays = ?)
    • Colonnes de jointure (JOIN ON t1.ref = t2.ref)
    • Colonnes de tri frequentes (ORDER BY date_traitement DESC)
    • Colonnes UNIQUE (auto-cree par la contrainte)

    Ne pas creer d'index sur :

    • Colonnes rarement utilisees en filtre
    • Tables tres petites (< 1000 lignes, le seq scan est plus rapide)
    • Colonnes avec tres peu de valeurs distinctes (BOOLEAN, STATUT avec 2 valeurs)
    • Colonnes modifiees tres frequemment (l'index doit etre mis a jour a chaque write)

    Le cout reel d'un index

    Un index n'est pas gratuit. Il a un cout en ecriture, en memoire et en maintenance. Creer des index sans discernement degrade les performances au lieu de les ameliorer.

    Cout en ecriture

    Chaque INSERT, UPDATE, DELETE doit mettre a jour tous les index de la table.

    INSERT sur une table avec 5 index :
→ 1 ecriture dans la table
→ 5 ecritures dans les index
= 6 operations d'ecriture au total

    Sur une table avec beaucoup d'ecritures (batch massif, logs, evenements), chaque index supplementaire ralentit directement les insertions.

    -- Exemple : mesurer l'impact
EXPLAIN ANALYZE INSERT INTO T_TRAITEMENT (...) VALUES (...);
-- Comparer le Planning Time et Execution Time avec et sans index

    Cout en memoire (shared_buffers)

    PostgreSQL garde les index en memoire (cache). Chaque index consomme de l'espace dans les shared_buffers.

    Trop d'index → les index les plus utiles sont evictes du cache → plus de lectures disque → ralentissement global.

    Cout en espace disque

    Un index B-Tree occupe en moyenne 10 a 30% de la taille de la table.

    -- Voir la taille des index
SELECT
    indexrelname AS index,
    pg_size_pretty(pg_relation_size(indexrelid)) AS taille
FROM pg_stat_user_indexes
WHERE relname = 'ma_table'
ORDER BY pg_relation_size(indexrelid) DESC;

    Cout en maintenance (VACUUM / AUTOVACUUM)

    Les lignes mortes (apres DELETE/UPDATE) restent dans les index jusqu'au VACUUM. Plus il y a d'index, plus le VACUUM est long.

    Sur des tables avec beaucoup de mises a jour, des index inutiles alourdissent considerablement la maintenance.

    La selectivite : le critere cle

    Un index est utile seulement si la colonne a une selectivite elevee • c'est-a-dire que la valeur filtree represente une petite fraction des lignes.

    -- BONNE selectivite : EMAIL est unique → index tres efficace
WHERE email = 'alice@example.com'  -- retourne 1 ligne sur 1 000 000

-- MAUVAISE selectivite : STATUT a 2 valeurs → index inutile
WHERE statut = 'ACTIF'  -- retourne 500 000 lignes sur 1 000 000

    Quand un index renvoie trop de lignes, PostgreSQL prefere le sequential scan • il est plus rapide que de suivre l'index puis lire chaque ligne individuellement.

    Regle pratique : un index est efficace si la requete retourne moins de 5 a 10% des lignes de la table.

    -- Verifier la selectivite d'une colonne
SELECT
    statut,
    COUNT(*) AS nb,
    ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER (), 2) AS pct
FROM T_TRAITEMENT
GROUP BY statut
ORDER BY nb DESC;

    Si une valeur represente 80% des lignes → l'index sur cette colonne sera rarement utilise par le planificateur.

    EXPLAIN ANALYZE : comprendre ce que fait PostgreSQL

    La commande la plus importante pour optimiser.

    EXPLAIN ANALYZE
SELECT * FROM T_INDIVIDU
WHERE STATUT = 'KO'
AND DATE_TRAITEMENT > '2024-01-01';

    Lire le resultat :

    Seq Scan on t_individu  (cost=0.00..245.00 rows=12 width=120)
                          ^^^^^^^^^^^^^^^^^  ^^^^
                          estimation cout    nb lignes estimees

Filter: ((statut = 'KO') AND (date_traitement > '2024-01-01'))
Rows Removed by Filter: 988

Planning Time: 0.5 ms
Execution Time: 45.2 ms

    Apres creation d'index : 

    Index Scan using idx_individu_ko on t_individu (cost=0.28..8.30 rows=12 width=120)
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
Index utilise !

Planning Time: 0.3 ms
Execution Time: 0.8 ms   <- 56x plus rapide

    Mots cles a surveiller :

    Mot cle Signification Bon ou Mauvais
    Seq Scan Lit toute la table Mauvais sur grande table
    Index Scan Utilise l'index, lit les lignes Bon
    Index Only Scan Repond depuis l'index seul Tres bon
    Bitmap Heap Scan Index + filtre en memoire Acceptable
    Nested Loop Jointure par boucle imbriquee Bon si petite table
    Hash Join Jointure par table de hachage Bon sur grandes tables
    Merge Join Jointure sur donnees triees Bon avec index

    Astuces et pieges courants

    1. LIKE avec % en debut de chaine n'utilise pas l'index

    -- N'utilise PAS l'index sur NOM_PRENOM
WHERE NOM_PRENOM LIKE '%MARTIN%'

-- Utilise l'index (prefixe fixe)
WHERE NOM_PRENOM LIKE 'MARTIN%'

    Pour la recherche libre, utiliser un index GIN avec pg_trgm : 

    CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_trgm_nom ON T_INDIVIDU USING GIN(NOM_PRENOM gin_trgm_ops);
-- Maintenant LIKE '%MARTIN%' utilise l'index

    2. Fonction sur colonne indexee desactive l'index

    -- N'utilise PAS l'index sur DATE_TRAITEMENT
WHERE DATE_TRUNC('month', DATE_TRAITEMENT) = '2024-01-01'

-- Utilise l'index
WHERE DATE_TRAITEMENT >= '2024-01-01' AND DATE_TRAITEMENT < '2024-02-01'

-- Ou creer un index fonctionnel
CREATE INDEX idx_month_traitement ON T_INDIVIDU(DATE_TRUNC('month', DATE_TRAITEMENT));

    3. Implicit cast desactive l'index

    -- La colonne NUM_REFERENCE est VARCHAR
-- N'utilise PAS l'index (cast implicite int -> varchar)
WHERE NUM_REFERENCE = 12345

-- Utilise l'index
WHERE NUM_REFERENCE = '12345'

    4. IS NULL et IS NOT NULL

    Les index B-Tree indexent les NULL depuis PostgreSQL 8.3.

    -- Utilise l'index
WHERE MESSAGE_ERREUR IS NULL
WHERE MESSAGE_ERREUR IS NOT NULL

    5. OR peut empecher l'utilisation de l'index

    -- Peut ne pas utiliser l'index
WHERE STATUT = 'KO' OR PAYS = 'FRANCE'

-- Preferer UNION pour forcer l'utilisation des index
SELECT * FROM T_INDIVIDU WHERE STATUT = 'KO'
UNION
SELECT * FROM T_INDIVIDU WHERE PAYS = 'FRANCE'

    Index et batch processing

    Dans un contexte batch (insertion/mise a jour massive), les index ont un cout : chaque INSERT/UPDATE doit mettre a jour tous les index de la table.

    Strategie pour les grosses insertions batch

    -- 1. Supprimer les index avant l'insertion massive
DROP INDEX IF EXISTS idx_individu_statut;
DROP INDEX IF EXISTS idx_individu_pays;

-- 2. Inserer en masse (beaucoup plus rapide sans index)
INSERT INTO T_INDIVIDU (...) VALUES (...), (...), ...;

-- 3. Recreer les index apres (PostgreSQL construit l'index en une passe)
CREATE INDEX CONCURRENTLY idx_individu_statut ON T_INDIVIDU(STATUT);
CREATE INDEX CONCURRENTLY idx_individu_pays   ON T_INDIVIDU(PAYS);

    CREATE INDEX CONCURRENTLY : construit l'index sans bloquer les lectures/ecritures. Plus lent mais sans downtime.

    UPSERT et index unique

    -- L'index UNIQUE sur NUM_REFERENCE permet le ON CONFLICT
INSERT INTO T_INDIVIDU (NUM_REFERENCE, STATUT, ...)
VALUES (?, ?, ...)
ON CONFLICT (NUM_REFERENCE) DO UPDATE SET
    STATUT = EXCLUDED.STATUT,
    DATE_TRAITEMENT = NOW();

    L'index unique est ici fonctionnel (contrainte metier) ET un mecanisme de performance (localisation rapide du conflit).

    Maintenance des index

    Identifier les index inutilises

    SELECT schemaname, tablename, indexname, idx_scan, idx_tup_read
FROM pg_stat_user_indexes
WHERE idx_scan = 0
ORDER BY pg_relation_size(indexrelid) DESC;

    Un index avec idx_scan = 0 n'est jamais utilise -> candidat a la suppression.

    Identifier les tables sans index sur FK

    SELECT conrelid::regclass AS table,
       conname AS contrainte,
       a.attname AS colonne
FROM pg_constraint c
JOIN pg_attribute a ON a.attnum = ANY(c.conkey) AND a.attrelid = c.conrelid
WHERE c.contype = 'f'
AND NOT EXISTS (
    SELECT 1 FROM pg_index i
    WHERE i.indrelid = c.conrelid
    AND a.attnum = ANY(i.indkey)
)
ORDER BY table;

    VACUUM et ANALYZE

    -- Met a jour les statistiques (aide le planificateur a choisir le bon plan)
ANALYZE T_INDIVIDU;

-- Recupere l'espace des lignes mortes (apres DELETE/UPDATE massif)
VACUUM T_INDIVIDU;

-- Les deux + recalcul index
VACUUM ANALYZE T_INDIVIDU;

    A executer apres une insertion/mise a jour batch massive.

    Exemple complet • table de traitement batch

    Index typiques pour une table utilisee dans un batch de traitement :

    -- Cle primaire (auto-cree)
-- Contrainte UNIQUE sur la reference metier (auto-cree)

-- Recherche par statut (rapport OK/KO)
CREATE INDEX idx_traitement_statut ON T_TRAITEMENT(STATUT);

-- Partial index : acces rapide aux erreurs (les plus consultees)
CREATE INDEX idx_traitement_ko ON T_TRAITEMENT(REFERENCE, DATE_TRAITEMENT)
WHERE STATUT = 'KO';

-- Analyse temporelle
CREATE INDEX idx_traitement_date ON T_TRAITEMENT(DATE_TRAITEMENT);

-- Index composite pour les requetes frequentes de rapport
CREATE INDEX idx_traitement_statut_date ON T_TRAITEMENT(STATUT, DATE_TRAITEMENT);

    Ces index peuvent etre geres via Liquibase ou Flyway :

    <changeSet id="add-indexes-traitement" author="dev">
    <createIndex tableName="T_TRAITEMENT" indexName="idx_traitement_statut">
        <column name="STATUT"/>
    </createIndex>
    <createIndex tableName="T_TRAITEMENT" indexName="idx_traitement_date">
        <column name="DATE_TRAITEMENT"/>
    </createIndex>
</changeSet>

    Flush • PostgreSQL et JPA

    Flush PostgreSQL (WAL / Checkpoint)

    PostgreSQL n'ecrit pas directement les donnees sur disque a chaque transaction. Il utilise un Write-Ahead Log (WAL) : les modifications sont d'abord ecrites dans le WAL (sequentiel, rapide), puis appliquees aux fichiers de donnees lors d'un checkpoint.

    INSERT/UPDATE/DELETE
       ↓
   WAL (disque, sequentiel)   ← rapide
       ↓
  shared_buffers (memoire)    ← modifie en memoire
       ↓
  checkpoint → fichiers data  ← ecriture asynchrone

    Flush PostgreSQL = forcer l'ecriture du WAL sur disque. Se produit a chaque COMMIT (par defaut synchronous_commit = on).

    Impact sur les index :

    Les index sont mis a jour dans les shared_buffers (memoire) en meme temps que les donnees. La persistance sur disque se fait au checkpoint. Pendant la fenetre entre le commit et le checkpoint, les donnees et index sont en memoire • recuperables apres crash grace au WAL.

    -- Forcer un checkpoint (admin seulement)
CHECKPOINT;

-- Verifier la frequence des checkpoints
SELECT * FROM pg_stat_bgwriter;

    Performance

    synchronous_commit = off desactive le flush WAL au commit • gain de performance mais risque de perte des dernieres transactions en cas de crash. A n'utiliser que pour des donnees non critiques (logs, metriques).

    Flush JPA / Hibernate

    Hibernate maintient un persistence context (cache de premier niveau) : les entites chargees ou modifiees sont gardees en memoire jusqu'au flush.

    Flush = synchronisation du persistence context vers la base de donnees (execution des SQL INSERT/UPDATE/DELETE).

    entityManager.persist(entity)   ← en memoire uniquement
       ↓
entityManager.flush()           ← SQL execute en base
       ↓
transaction.commit()            ← commit + WAL flush PostgreSQL

    Modes de flush :

    Mode Comportement
    AUTO (defaut) Flush automatique avant une requete JPQL/HQL qui concerne les entites modifiees
    COMMIT Flush uniquement au commit
    MANUAL Flush uniquement sur appel explicite flush() 
    // Changer le mode de flush
entityManager.setFlushMode(FlushModeType.COMMIT);

// Flush explicite
entityManager.flush();

// Vider le cache de premier niveau
entityManager.clear();

    Impact sur les index dans un contexte Spring Batch :

    En mode AUTO, Hibernate peut flusher en cours de step si une requete JPQL touche les entites en cours de traitement. Cela peut :

    • Declencher des verifications de contraintes d'unicite (index UNIQUE)
    • Creer des verrous inattendus
    • Degrader les performances si le flush se produit trop souvent
    // Pattern recommande en Spring Batch avec JPA
@Component
@StepScope
public class MonProcessor implements ItemProcessor<Input, Output> {

    @PersistenceContext
    private EntityManager em;

    @Override
    public Output process(Input item) {
        // Flush explicite seulement si necessaire
        // Sinon laisser Spring Batch gerer via @Transactional sur le chunk
        return transformer(item);
    }
}

    Flush et index : le point cle

    Le flush JPA execute le SQL → PostgreSQL met a jour les index en memoire → le checkpoint persist sur disque. La chaine est :

    flush() → SQL → index mis a jour en shared_buffers → checkpoint → disque

    Un flush trop frequent (mode AUTO avec beaucoup de requetes) = beaucoup de mises a jour d'index en cours de transaction = contention + ralentissement.
