    AOP - Programmation Orientée Aspect avec Spring

    Pourquoi l'AOP existe

    Dans une application backend, certains comportements traversent toutes les couches sans appartenir à aucune en particulier :

    • la gestion des exceptions
    • le logging des entrées/sorties
    • les transactions
    • la sécurité (contrôle d'accès)
    • le cache
    • les métriques

    Ces comportements s'appellent des cross-cutting concerns - des préoccupations transversales.

    Sans AOP, on les duplique dans chaque méthode :

    public BookingView createBooking(CreateBookingRequest request) {
    log.info("createBooking called");           // logging dupliqué
    checkPermission();                           // sécurité dupliquée
    try {
        // logique métier
    } catch (Exception e) {
        log.error("error", e);                  // gestion erreur dupliquée
        throw e;
    }
}

    L'AOP extrait ces comportements en dehors du code métier.

    Vocabulaire fondamental

    Aspect

    Un Aspect regroupe la logique transversale. Exemple : un aspect de logging, un aspect de sécurité.

    Advice

    Un Advice est le code à exécuter. Il existe plusieurs types :

    Type Quand Usage typique
    @Before Avant la méthode Vérification d'accès, log d'entrée
    @After Après la méthode (toujours) Nettoyage, log de sortie
    @AfterReturning Après si succès Log du résultat
    @AfterThrowing Après si exception Log de l'erreur
    @Around Enveloppe la méthode Timing, retry, transaction

    Pointcut

    Un Pointcut est l'expression qui définit où l'aspect s'applique.

    // S'applique à toutes les méthodes des controllers
@Pointcut("within(@org.springframework.web.bind.annotation.RestController *)")
public void restControllerMethods() {}

// S'applique à toutes les méthodes du package application
@Pointcut("execution(* dev.kore.hexagonal.application..*(..))")
public void applicationLayer() {}

    JoinPoint

    Un JoinPoint est un point précis d'exécution où l'aspect s'insère - en pratique, l'appel d'une méthode.

    Weaving

    Le Weaving est l'opération qui relie l'aspect au code cible. Spring utilise des proxies dynamiques - l'objet réel est enveloppé dans un proxy qui intercepte les appels.

    Comment Spring AOP fonctionne

    Spring AOP est basé sur des proxies. Quand Spring crée un bean, il peut l'envelopper dans un proxy qui intercepte les appels de méthodes.

    [Controller]
     |
     v
[Proxy Spring AOP]  ←── les aspects s'exécutent ici
     |
     v
[Service réel]

    C'est pour ça que :

    • @Transactional fonctionne sur des appels externes au bean, pas sur des appels internes
    • @ControllerAdvice intercepte les exceptions des controllers (qui passent par le dispatcher servlet de Spring)

    Limite importante

    @Service
public class BookingService {

    public void createBooking(...) {
        this.validateAndSave(...);  // appel interne - le proxy est contourné
    }

    @Transactional  // n'aura AUCUN EFFET ici
    public void validateAndSave(...) { }
}

    Un appel interne (this.method()) court-circuite le proxy. L'AOP ne s'applique pas.

    @ControllerAdvice - l'AOP pour les exceptions HTTP

    @RestControllerAdvice est le mécanisme AOP de Spring pour centraliser la gestion des exceptions HTTP.

    Au lieu de gérer les erreurs dans chaque controller, Spring intercepte toutes les exceptions non catchées et les route vers le handler approprié.

    Implémentation dans kore-hexagonal

    @RestControllerAdvice
public class GlobalExceptionHandler {

    // BookingNotFoundException → 404
    @ExceptionHandler(BookingNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(
            BookingNotFoundException ex, HttpServletRequest request) {
        return ResponseEntity.status(404).body(new ErrorResponse(
                ZonedDateTime.now(), 404,
                "Introuvable", ex.getMessage(),
                request.getRequestURI(), null
        ));
    }

    // Règle métier violée → 400
    @ExceptionHandler(BookingDomainException.class)
    public ResponseEntity<ErrorResponse> handleDomain(
            BookingDomainException ex, HttpServletRequest request) {
        return ResponseEntity.badRequest().body(new ErrorResponse(
                ZonedDateTime.now(), 400,
                "Règle métier violée", ex.getMessage(),
                request.getRequestURI(), null
        ));
    }

    // Filet de sécurité - toute exception non prévue → 500
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneric(
            Exception ex, HttpServletRequest request) {
        return ResponseEntity.internalServerError().body(new ErrorResponse(
                ZonedDateTime.now(), 500,
                "Erreur interne", "Une erreur inattendue s'est produite",
                request.getRequestURI(), null
        ));
    }
}

    Ce que Spring fait en coulisse : 1. le controller lève une BookingNotFoundException 2. Spring MVC intercepte l'exception (via AOP / DispatcherServlet) 3. Spring cherche un @ExceptionHandler compatible dans le @RestControllerAdvice 4. Il trouve handleNotFound et l'exécute 5. La réponse 404 est retournée au client

    Le code du controller ne sait pas qu'un handler existe - il lève simplement l'exception.

    NullPointerException - cas particulier

    La NPE est une RuntimeException non vérifiée. Elle peut être interceptée par le @ExceptionHandler(Exception.class) - mais ce n'est pas la bonne stratégie.

    Pourquoi la NPE ne doit pas atteindre le handler

    Une NPE qui arrive jusqu'au GlobalExceptionHandler signifie qu'il y a un bug dans le code - une variable null non anticipée. Le handler la transforme en 500, mais le vrai problème n'est pas résolu.

    Trois niveaux de défense

    Niveau 1 - Validation en entrée (avant tout traitement)

    @PostMapping("/bookings")
public ResponseEntity<BookingView> create(
        @Valid @RequestBody CreateBookingRequest request) {
    // @Valid déclenche la validation Bean Validation
    // MethodArgumentNotValidException si invalide → géré par le handler
}

    Niveau 2 - Design défensif dans le domaine

    // Mauvais
public Booking(String assetRef, LocalDate startDate, LocalDate endDate) {
    this.assetRef = assetRef; // peut être null → NPE plus tard
}

// Correct
public Booking(String assetRef, LocalDate startDate, LocalDate endDate) {
    if (assetRef == null || assetRef.isBlank()) {
        throw new BookingDomainException("assetRef obligatoire");
    }
    this.assetRef = assetRef;
}

    Niveau 3 - Optional pour les lectures

    // Mauvais - NPE si absent
Booking booking = repository.findById(id).get();

// Correct - exception métier explicite
Booking booking = repository.findById(id)
        .orElseThrow(() -> new BookingNotFoundException(
            "Booking introuvable : " + id));

    Résumé NPE

    Contexte Approche
    Données entrantes (HTTP) @Valid + @NotNull sur le DTO
    Construction du domaine Validation explicite dans le constructeur
    Lecture repository Optional.orElseThrow() avec exception métier
    Données externes (API, fichier) Validation défensive avant usage

    Une NPE en production est toujours un bug - jamais un cas métier attendu.

    Aspect custom - exemple de logging transversal

    Quand @ControllerAdvice ne suffit pas (logging de toutes les méthodes d'un service, métriques d'exécution), on crée un @Aspect custom.

    @Aspect
@Component
@Slf4j
public class ApplicationLayerLoggingAspect {

    // S'applique à toutes les méthodes des handlers applicatifs
    @Pointcut("execution(* dev.kore.hexagonal.application..handler.*.*(..))")
    public void handlerMethods() {}

    @Around("handlerMethods()")
    public Object logExecution(ProceedingJoinPoint joinPoint) throws Throwable {
        String method = joinPoint.getSignature().toShortString();
        long start = System.currentTimeMillis();

        try {
            Object result = joinPoint.proceed();
            long duration = System.currentTimeMillis() - start;
            log.info("{} completed in {}ms", method, duration);
            return result;

        } catch (Exception e) {
            log.error("{} failed : {}", method, e.getMessage());
            throw e;  // on re-lève pour ne pas avaler l'exception
        }
    }
}

    Ce que cet aspect fait sans toucher au code des handlers : - logue le nom de la méthode appelée - mesure le temps d'exécution - logue les erreurs avec leur contexte

    AOP dans Spring - exemples natifs

    Spring utilise massivement l'AOP pour ses propres mécanismes :

    Annotation Mécanisme AOP Ce qu'elle fait
    @Transactional @Around Ouvre/ferme une transaction autour de la méthode
    @Cacheable @Around Retourne la valeur du cache si disponible
    @Async @Around Exécute la méthode dans un thread séparé
    @PreAuthorize @Before Vérifie les droits avant l'exécution
    @RestControllerAdvice Exception handler AOP Intercepte les exceptions non catchées

    Synthèse

    • AOP = extraire les comportements transversaux hors du code métier
    • Spring AOP = basé sur des proxies dynamiques • ne fonctionne pas sur les appels internes
    • @RestControllerAdvice = l'AOP de Spring pour centraliser la gestion des exceptions HTTP
    • NPE = toujours un bug à prévenir en amont, pas à gérer dans le handler global
    • @Aspect custom = quand on veut appliquer un comportement transversal à une couche entière
