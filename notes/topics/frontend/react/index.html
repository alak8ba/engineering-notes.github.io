    React • Fondamentaux & Hooks

    Positionnement

    React est dominant dans les startups et le secteur tech. Les concepts ci-dessous couvrent les questions d'entretien React/TypeScript les plus fréquentes.

    Composants fonctionnels

    Depuis React 16.8, tout se fait avec des fonctions. Les class components sont obsolètes.

    interface Props {
  userId: string;
  onSelect?: (user: User) => void;
}

const UserCard: React.FC<Props> = ({ userId, onSelect }) => {
  const [user, setUser] = useState<User | null>(null);

  useEffect(() => {
    fetchUser(userId).then(setUser);
  }, [userId]);

  if (!user) return <Spinner />;

  return (
    <div onClick={() => onSelect?.(user)}>
      <h2>{user.name}</h2>
    </div>
  );
};

    Hooks fondamentaux

    useState

    const [count, setCount] = useState(0);
const [user, setUser] = useState<User | null>(null);

// Mise à jour fonctionnelle (recommandée quand le nouvel état dépend de l'ancien)
setCount(prev => prev + 1);

// Mise à jour d'objet • toujours créer une nouvelle référence
setUser(prev => ({ ...prev!, name: 'Alice' }));

    useEffect

    // S'exécute après chaque render
useEffect(() => { ... });

// S'exécute une seule fois (mount)
useEffect(() => { ... }, []);

// S'exécute quand userId change
useEffect(() => {
  fetchUser(userId).then(setUser);
}, [userId]);

// Cleanup • s'exécute avant le prochain effet ou au unmount
useEffect(() => {
  const sub = eventBus.subscribe(handler);
  return () => sub.unsubscribe();   // cleanup
}, []);

    Erreur classique useEffect

    Oublier les dépendances ou en ajouter de trop. La règle : tout ce qui est utilisé à l'intérieur de l'effet doit être dans le tableau de dépendances.

    useRef

    // Accès DOM
const inputRef = useRef<HTMLInputElement>(null);
inputRef.current?.focus();

// Valeur persistante sans re-render (timers, abonnements)
const timerRef = useRef<NodeJS.Timeout>();
timerRef.current = setTimeout(() => ..., 1000);

    useContext

    // Création du contexte
const AuthContext = createContext<AuthContextType | null>(null);

// Hook d'accès custom (recommandé)
export const useAuth = () => {
  const ctx = useContext(AuthContext);
  if (!ctx) throw new Error('useAuth doit être dans AuthProvider');
  return ctx;
};

// Provider
export const AuthProvider: React.FC<{ children: ReactNode }> = ({ children }) => {
  const [user, setUser] = useState<User | null>(null);
  return (
    <AuthContext.Provider value={{ user, setUser }}>
      {children}
    </AuthContext.Provider>
  );
};

// Consommation
const { user } = useAuth();

    useReducer

    type Action =
  | { type: 'SET_LOADING' }
  | { type: 'SET_DATA'; payload: User[] }
  | { type: 'SET_ERROR'; payload: string };

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'SET_LOADING': return { ...state, loading: true };
    case 'SET_DATA':    return { loading: false, data: action.payload, error: null };
    case 'SET_ERROR':   return { loading: false, data: [], error: action.payload };
  }
}

const [state, dispatch] = useReducer(reducer, { loading: false, data: [], error: null });

dispatch({ type: 'SET_LOADING' });

    useState vs useReducer

    useState pour un état simple indépendant. useReducer quand plusieurs valeurs d'état évoluent ensemble ou quand la logique de transition est complexe.

    Performance • useMemo & useCallback

    useCallback • mémoriser une fonction

    // Sans useCallback : nouvelle référence à chaque render → re-render de l'enfant
// Avec useCallback : même référence si les dépendances n'ont pas changé
const handleDelete = useCallback((id: string) => {
  setUsers(prev => prev.filter(u => u.id !== id));
}, []);   // pas de dépendances → référence stable

    useMemo • mémoriser un calcul

    const filteredUsers = useMemo(
  () => users.filter(u => u.role === selectedRole),
  [users, selectedRole]   // recalcule seulement si ces valeurs changent
);

    React.memo • éviter le re-render d'un composant

    const UserCard = React.memo(({ user, onDelete }: Props) => {
  return <div>{user.name}</div>;
});
// Ne re-render que si user ou onDelete changent de référence

    Optimisation prématurée

    Ne pas ajouter useMemo/useCallback partout. Les utiliser uniquement quand un composant enfant re-render inutilement ou pour des calculs vraiment coûteux.

    Custom Hooks

    Extraire la logique réutilisable dans des hooks prefixés use :

    function useFetch<T>(url: string) {
  const [data, setData] = useState<T | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    let cancelled = false;
    setLoading(true);

    fetch(url)
      .then(r => r.json())
      .then(d => { if (!cancelled) setData(d); })
      .catch(e => { if (!cancelled) setError(e); })
      .finally(() => { if (!cancelled) setLoading(false); });

    return () => { cancelled = true; };
  }, [url]);

  return { data, loading, error };
}

// Usage
const { data: users, loading } = useFetch<User[]>('/api/users');

    État complexe / transitions useReducer

    Cache serveur + état async React Query (TanStack Query)

    Patterns courants

    Composant contrôlé vs non contrôlé

    // Contrôlé • React gère la valeur (recommandé)
const [value, setValue] = useState('');
<input value={value} onChange={e => setValue(e.target.value)} />

// Non contrôlé • le DOM gère la valeur
const ref = useRef<HTMLInputElement>(null);
<input ref={ref} defaultValue="initial" />
// lire: ref.current?.value

    Compound Components

    // Pattern pour des composants liés (ex: Select + Option)
const Select = ({ children, onChange }: SelectProps) => {
  const [selected, setSelected] = useState<string>();
  return (
    <SelectContext.Provider value={{ selected, onChange: setSelected }}>
      <div>{children}</div>
    </SelectContext.Provider>
  );
};

Select.Option = ({ value, label }: OptionProps) => {
  const { onChange } = useContext(SelectContext)!;
  return <div onClick={() => onChange(value)}>{label}</div>;
};

// Usage
<Select onChange={setRole}>
  <Select.Option value="admin" label="Admin" />
  <Select.Option value="user" label="User" />
</Select>

    Error Boundary

    class ErrorBoundary extends React.Component<Props, { hasError: boolean }> {
  state = { hasError: false };

  static getDerivedStateFromError() {
    return { hasError: true };
  }

  componentDidCatch(error: Error, info: ErrorInfo) {
    console.error(error, info);
  }

  render() {
    if (this.state.hasError) return <FallbackUI />;
    return this.props.children;
  }
}

    Cas concrets

    Pourquoi ne pas muter le state directement ?

    React compare les références pour décider de re-render. Si on mute directement (state.name = 'X'), la référence ne change pas → pas de re-render. Toujours créer une nouvelle référence : setState({ ...state, name: 'X' }).

    Virtual DOM ?

    Représentation légère du DOM en mémoire. À chaque changement d'état, React calcule le diff entre l'ancien et le nouveau Virtual DOM et applique seulement les modifications nécessaires au vrai DOM (reconciliation).

    useEffect sans tableau vs [] vs [dep] ?
    • Sans tableau : s'exécute après chaque render
    • [] vide : une seule fois au mount
    • [dep] : au mount + à chaque changement de dep
    Quand utiliser useCallback ?

    Quand une fonction est passée en prop à un composant mémorisé (React.memo). Sans useCallback, nouvelle référence à chaque render → React.memo ne protège plus. Ne pas l'utiliser sans mesurer d'abord.

    Context vs Zustand vs Redux ?

    Context : natif, adapté aux états qui changent peu (thème, auth, langue). Zustand : léger, performant, DevTools. Redux Toolkit : états complexes à grande échelle. Context avec updates fréquents → re-renders en cascade.

    Réconciliation (reconciliation) ?

    Algorithme React pour déterminer le diff entre deux arbres de composants. La key est critique dans les listes : elle permet à React d'identifier les éléments et d'éviter de recréer ce qui n'a pas changé.

    Points clés

    • tout se fait avec des fonctions depuis React 16.8 • les class components sont obsolètes
    • useState + useEffect couvrent 80% des cas
    • ne jamais muter le state directement • toujours créer une nouvelle référence
    • useCallback et useMemo : optimisation ciblée, pas systématique
    • Custom hooks : extraire la logique réutilisable préfixée use
    • React Query pour le cache HTTP • Zustand pour l'état global
