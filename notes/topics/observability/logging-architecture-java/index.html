    Architecture de Logging dans les Applications Java

    Note de contexte (12 mai 2026) : cette note constitue un retour d'expérience suite à des échanges avec des experts du domaine. Elle est rédigée le jour même et vise à structurer les notions abordées pour consolidation et montée en compétence.

    Introduction

    Le logging est une composante essentielle de l'ingénierie logicielle orientée production.

    Dans les systèmes backend modernes, les logs ne servent pas uniquement au debug. Ils sont indispensables pour :

    • la visibilité runtime
    • l'investigation d'incidents
    • le tracing distribué
    • le support production
    • la détection de comportements anormaux

    Une architecture de logging bien conçue permet de comprendre ce que fait réellement le système pendant son exécution • sans avoir à redéployer ou attacher un debugger.

    Architecture en trois couches

    Code applicatif
      |
   SLF4J          ← façade (API)
      |
   Logback        ← implémentation (Spring Boot default)
      |
Console / Fichier / Loki / ELK

    Cette séparation permet de changer d'implémentation sans toucher au code.

    SLF4J • la façade standard

    SLF4J (Simple Logging Facade for Java) est l'API de logging standard. Le code applicatif ne dépend que de SLF4J, jamais de l'implémentation concrète.

    Déclaration classique

    import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class MonService {
    private static final Logger log = LoggerFactory.getLogger(MonService.class);
}

    Avec Lombok @Slf4j

    Dans les projets Spring Boot modernes, @Slf4j génère automatiquement le logger :

    import lombok.extern.slf4j.Slf4j;

@Slf4j
@Component
public class AssureItemProcessor implements ItemProcessor<AssureDto, AssureResultDto> {

    @Override
    public AssureResultDto process(AssureDto assure) {
        log.debug("Traitement individu ref={}", assure.getReference());
        // ...
    }
}

    @Slf4j évite le boilerplate • c'est équivalent à déclarer private static final Logger log = LoggerFactory.getLogger(AssureItemProcessor.class).

    Logback • l'implémentation par défaut

    Logback est inclus automatiquement via spring-boot-starter-logging. Il gère :

    • l'écriture des logs (console, fichier, Loki...)
    • la rotation des fichiers
    • les formats de sortie (texte ou JSON)
    • le logging asynchrone

    Configuration des niveaux dans application.yml

    logging:
  level:
    dev.kore.batch: INFO       # tout le code applicatif en INFO
    org.springframework.batch: INFO

    En développement on peut passer à DEBUG pour voir le détail. En production, INFO est le niveau standard.

    Configuration Logback (logback-spring.xml)

    <configuration>

  <!-- Appender console en JSON pour Grafana Loki / ELK -->
  <appender name="JSON" class="ch.qos.logback.core.ConsoleAppender">
    <encoder class="net.logstash.logback.encoder.LogstashEncoder">
      <includeMdcKeyName>correlationId</includeMdcKeyName>
      <includeMdcKeyName>jobName</includeMdcKeyName>
    </encoder>
  </appender>

  <!-- Appender console texte pour le développement local -->
  <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
    <encoder>
      <pattern>%d{HH:mm:ss} %-5level [%thread] %logger{36} - %msg%n</pattern>
    </encoder>
  </appender>

  <springProfile name="prod">
    <root level="INFO"><appender-ref ref="JSON"/></root>
  </springProfile>

  <springProfile name="dev,local">
    <root level="DEBUG"><appender-ref ref="CONSOLE"/></root>
  </springProfile>

</configuration>

    Niveaux de logs

    Niveau Usage Exemple
    TRACE Analyse interne très fine Valeur d'une variable à chaque itération
    DEBUG Développement, suivi item par item log.debug("Traitement ref={}", ref)
    INFO Événements normaux en production Début/fin de job, actions métier
    WARN Comportement inattendu non bloquant Erreur fonctionnelle, doublon ignoré
    ERROR Erreur impactant le système Exception technique, service indisponible

    Logging structuré avec contexte

    Exemple simple avec paramètres

    // Mauvais : concaténation String (coûteux si log désactivé)
log.info("Traitement de " + assure.getReference() + " OK");

// Correct : paramètres lazy • la String n'est construite que si le niveau est actif
log.info("Traitement ref={} OK", assure.getReference());

    Logging structuré avec StructuredArguments (Logstash)

    Dans kore-batch, le processor utilise StructuredArguments pour produire des logs exploitables par Grafana Loki ou ELK :

    import net.logstash.logback.argument.StructuredArguments;

log.warn("Erreur fonctionnelle [{}] : {}",
    StructuredArguments.value("reference", assure.getReference()),
    e.getMessage());

    En sortie JSON :

    {
  "level": "WARN",
  "message": "Erreur fonctionnelle",
  "reference": "5000012345",
  "error": "Identifiant invalide (doit faire 10 caracteres)"
}

    Le champ reference devient un champ JSON indexé • requêtable directement dans Grafana ou Kibana.

    Logging dans les listeners Spring Batch

    Le pattern listeners est la bonne approche pour logguer les événements de cycle de vie d'un batch sans polluer le code métier.

    Listener de job • kore-batch

    @Slf4j
@Component
public class BatchJobExecutionListener implements JobExecutionListener {

    @Override
    public void beforeJob(JobExecution jobExecution) {
        log.info("=== DEBUT DU BATCH [{}] - JobId={} ===",
            jobExecution.getJobInstance().getJobName(),
            jobExecution.getJobId());
    }

    @Override
    public void afterJob(JobExecution jobExecution) {
        Duration duree = Duration.between(
            jobExecution.getStartTime(), jobExecution.getEndTime());

        log.info("=== FIN DU BATCH [{}] - Statut={} - Duree={}s ===",
            jobExecution.getJobInstance().getJobName(),
            jobExecution.getStatus(),
            duree.toSeconds());

        ISynthese synthese = (ISynthese)
            jobExecution.getExecutionContext().get(ISynthese.SYNTHESE_KEY);

        if (synthese != null) {
            log.info("SYNTHESE : total={}, OK={}, KO={}, errTech={}, errFonc={}, doublons={}",
                synthese.getTotal(), synthese.getNbOK(), synthese.getNbKO(),
                synthese.getNbErreursTechniques(), synthese.getNbErreursFonctionnelles(),
                synthese.getNbDoublons());
        }
    }
}

    Listener de step • avec détail par partition

    @Slf4j
@Component
public class TraitementAssuresStepListener implements StepExecutionListener {

    @Override
    public void beforeStep(StepExecution stepExecution) {
        log.info("=== DEBUT DU STEP [{}] ===", stepExecution.getStepName());
    }

    @Override
    public ExitStatus afterStep(StepExecution stepExecution) {
        IndividuSyntheseDto synthese = (IndividuSyntheseDto)
            stepExecution.getExecutionContext().get(ISynthese.SYNTHESE_KEY);

        if (synthese != null) {
            log.info("=== FIN DU STEP [{}] - lu={}, OK={}, KO={}, errFonc={} ===",
                stepExecution.getStepName(),
                stepExecution.getReadCount(),
                synthese.getNbOK(),
                synthese.getNbKO(),
                synthese.getNbErreursFonctionnelles());

            if (!synthese.getReferencesEnErreur().isEmpty()) {
                log.warn("References en erreur : {}", synthese.getReferencesEnErreur());
            }
        }
        return stepExecution.getExitStatus();
    }
}

    Ce que ces logs permettent en production : - savoir immédiatement si le batch a réussi ou échoué - voir la durée d'exécution - connaître le nombre d'items traités, OK, KO - identifier les références en erreur sans ouvrir la base

    MDC • Mapped Diagnostic Context

    Le MDC permet d'injecter automatiquement du contexte dans tous les logs produits pendant un traitement.

    import org.slf4j.MDC;

// Injection du contexte
MDC.put("correlationId", UUID.randomUUID().toString());
MDC.put("jobName", "traitement-individus");

// Tous les logs suivants contiennent automatiquement ces champs
log.info("Debut du traitement");  // → { "correlationId": "abc-123", "jobName": "..." }

// Nettoyage obligatoire (risque de fuite mémoire sinon)
MDC.clear();

    Dans un contexte Spring, le MDC est souvent alimenté dans un filtre HTTP ou un JobExecutionListener.

    Stratégie de logging par couche

    Couche Niveau recommandé Ce qu'on logue
    Controller / Handler INFO en entrée, ERROR si 500 Requête reçue, réponse envoyée
    Service / Domain Pas de log sauf événement significatif Action métier importante
    Processor (batch) DEBUG item par item, WARN si KO Référence traitée, erreur fonctionnelle
    Listener (batch) INFO début/fin, WARN si erreurs Synthèse complète
    Infrastructure ERROR si indisponible Base down, Kafka unreachable

    Bonnes pratiques

    • utiliser @Slf4j (Lombok) • jamais System.out.println
    • préférer les paramètres {} à la concaténation String
    • ne jamais logger de données sensibles (mot de passe, numéro de carte, NIR)
    • logguer à la frontière du système, pas dans chaque couche
    • en production : niveau INFO par défaut, DEBUG interdit sauf investigation ponctuelle
    • utiliser le logging structuré JSON en production pour les outils de centralisation

    Transition vers l'observabilité complète

    Le logging est le premier pilier. Les trois ensemble permettent une visibilité totale :

    Pilier Outil Question à laquelle il répond
    Logs Logback + Loki/ELK Que s'est-il passé exactement ?
    Métriques Micrometer + Prometheus Le système est-il en bonne santé ?
    Tracing OpenTelemetry + Jaeger Quel chemin a suivi cette requête ?
