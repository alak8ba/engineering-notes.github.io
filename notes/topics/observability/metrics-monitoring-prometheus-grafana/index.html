    Metrics, Actuator, Prometheus & Grafana dans Spring Boot

    Note de contexte (12 mai 2026) : cette note constitue un retour d'expérience suite à des échanges avec des experts du domaine. Elle est rédigée le jour même et vise à structurer les notions abordées pour consolidation et montée en compétence.

    Introduction

    Les métriques sont un pilier essentiel de l'observabilité. Contrairement aux logs (événements textuels) et aux traces (parcours de requêtes), les métriques représentent des données numériques agrégées sur l'état du système.

    Elles répondent à des questions comme :

    • le système est-il en bonne santé en ce moment ?
    • combien de requêtes par seconde traite-t-on ?
    • quel est le taux d'erreur ?
    • la JVM a-t-elle assez de mémoire ?

    La chaîne complète

    Application Spring Boot
        |
   Micrometer        ← bibliothèque de métriques (abstraction)
        |
Spring Boot Actuator ← exposition via /actuator/prometheus
        |
   Prometheus        ← scraping périodique + stockage
        |
    Grafana          ← visualisation, dashboards, alertes

    Chaque composant a un rôle précis. On ne peut pas remplacer l'un sans impacter les autres.

    Spring Boot Actuator

    Actuator expose des endpoints HTTP pour observer l'état interne de l'application.

    Configuration

    management:
  endpoints:
    web:
      exposure:
        include: health, prometheus, metrics, info
  metrics:
    tags:
      application: ${spring.application.name}
      environment: ${spring.profiles.active:default}

    Endpoints principaux

    Endpoint Rôle
    /actuator/health Etat global (UP/DOWN) + dépendances
    /actuator/prometheus Métriques au format Prometheus
    /actuator/metrics Liste et valeurs des métriques
    /actuator/info Informations applicatives

    Réponse /actuator/health

    {
  "status": "UP",
  "components": {
    "db": { "status": "UP" },
    "diskSpace": { "status": "UP" },
    "kafka": { "status": "UP" }
  }
}

    Health Checks • pattern kore-batch

    Spring Boot Actuator fournit des health checks pour les applications web. En contexte batch, le socle kore-batch implémente un pattern de vérification avant le lancement du job • il ne sert à rien de démarrer un traitement si la base est inaccessible.

    HealthResult • résultat d'un check

    public class HealthResult {

    private final String name;
    private final HealthStatus status;  // UP ou DOWN
    private final String detail;

    public static HealthResult up(String name) {
        return new HealthResult(name, HealthStatus.UP, null);
    }

    public static HealthResult down(String name, String detail) {
        return new HealthResult(name, HealthStatus.DOWN, detail);
    }

    public boolean isUp() { return HealthStatus.UP.equals(status); }

    @Override
    public String toString() {
        return detail != null
            ? "[%s] %s - %s".formatted(status, name, detail)
            : "[%s] %s".formatted(status, name);
    }
}

    DatabaseHealthIndicator • check base de données

    @Slf4j
@Component
@RequiredArgsConstructor
public class DatabaseHealthIndicator implements BatchHealthIndicator {

    private final DataSource dataSource;

    @Override
    public HealthResult check() {
        try (Connection connection = dataSource.getConnection()) {
            boolean valid = connection.isValid(3);  // timeout 3 secondes
            if (valid) {
                log.info("Health [DATABASE] : UP");
                return HealthResult.up("DATABASE");
            }
            log.warn("Health [DATABASE] : DOWN - connexion invalide");
            return HealthResult.down("DATABASE", "connexion invalide");
        } catch (Exception e) {
            log.error("Health [DATABASE] : DOWN - {}", e.getMessage());
            return HealthResult.down("DATABASE", e.getMessage());
        }
    }
}

    AbstractWSHealthIndicator • template pour services externes

    Pour chaque service externe (API REST, répertoire réseau, WS), le projet métier étend ce template :

    @Slf4j
public abstract class AbstractWSHealthIndicator implements BatchHealthIndicator {

    protected abstract String getName();
    protected abstract boolean ping();

    @Override
    public HealthResult check() {
        try {
            boolean up = ping();
            if (up) {
                log.info("Health [{}] : UP", getName());
                return HealthResult.up(getName());
            }
            log.warn("Health [{}] : DOWN - ping KO", getName());
            return HealthResult.down(getName(), "ping KO");
        } catch (Exception e) {
            log.error("Health [{}] : DOWN - {}", getName(), e.getMessage());
            return HealthResult.down(getName(), e.getMessage());
        }
    }
}

    Implémentation concrète dans kore-batch-sample :

    @Component
public class FichierSourceHealthIndicator extends AbstractWSHealthIndicator {

    @Value("${batch.health.source-directory}")
    private String sourceDirectory;

    @Override
    protected String getName() { return "FICHIER_SOURCE"; }

    @Override
    protected boolean ping() {
        File dir = new File(sourceDirectory);
        return dir.exists() && dir.isDirectory() && dir.canRead();
    }
}

    BatchHealthAggregator • orchestration des checks

    @Slf4j
@Component
@RequiredArgsConstructor
public class BatchHealthAggregator {

    private final List<BatchHealthIndicator> indicators;
    // Spring injecte automatiquement tous les beans BatchHealthIndicator

    public boolean checkAll() {
        log.info("=== DIAGNOSTIC ENVIRONNEMENT ===");

        boolean allUp = true;
        for (BatchHealthIndicator indicator : indicators) {
            HealthResult result = indicator.check();
            log.info("  {}", result);
            if (!result.isUp()) {
                allUp = false;
            }
        }

        if (allUp) {
            log.info("=== ENVIRONNEMENT OK - lancement du batch ===");
        } else {
            log.error("=== ENVIRONNEMENT KO - batch annule ===");
        }
        return allUp;
    }
}

    Intégration dans le BatchLauncher

    @Override
public void run(String... args) {
    int codeRetour = CODE_ERREUR_TECHNIQUE;
    try {
        // Vérification de l'environnement avant tout lancement
        if (!healthAggregator.checkAll()) {
            log.error("Health check KO - batch annule");
            System.exit(CODE_ERREUR_TECHNIQUE);
            return;
        }

        // Lancement du job uniquement si tous les checks sont UP
        JobExecution jobExecution = jobLauncher.run(job, jobParameters);
        codeRetour = resolveExitCode(jobExecution);

    } catch (TechnicalException e) {
        log.error("Erreur technique : {}", e.getMessage(), e);
    } finally {
        System.exit(codeRetour);
    }
}

    Ce que ce pattern garantit : le batch ne démarre jamais dans un environnement dégradé. Le code retour -1 permet au scheduler (cron, Control-M) de détecter l'échec et d'alerter.

    Micrometer • métriques custom

    Micrometer est la couche d'abstraction entre le code et le système de monitoring. Il fonctionne comme SLF4J pour le logging • le code écrit des métriques via Micrometer, l'implémentation (Prometheus, Datadog, etc.) est interchangeable.

    Types de métriques

    Type Usage Exemple
    Counter Compteur incrémental Nombre de bookings créés
    Gauge Valeur instantanée Taille d'une queue
    Timer Durée d'exécution Temps de traitement d'un item
    DistributionSummary Distribution statistique Taille des payloads

    Counter • compter des événements métier

    @Component
@RequiredArgsConstructor
public class BookingMetrics {

    private final MeterRegistry registry;

    public void incrementCreated() {
        Counter.builder("booking.created")
                .description("Nombre de bookings créés")
                .tag("status", "success")
                .register(registry)
                .increment();
    }

    public void incrementFailed(String reason) {
        Counter.builder("booking.created")
                .description("Nombre de bookings en erreur")
                .tag("status", "error")
                .tag("reason", reason)
                .register(registry)
                .increment();
    }
}

    Timer • mesurer des durées

    Timer timer = Timer.builder("booking.processing.duration")
        .description("Durée de traitement d'un booking")
        .register(registry);

timer.record(() -> {
    // code à mesurer
    processBooking(booking);
});

    Gauge • valeur instantanée

    // Exposer la taille d'une queue en temps réel
Gauge.builder("batch.queue.size", queue, Collection::size)
        .description("Nombre d'items en attente de traitement")
        .register(registry);

    Prometheus • scraping et stockage

    Prometheus scrute périodiquement le endpoint /actuator/prometheus de chaque application.

    Modèle Pull

    Prometheus ──(scrape toutes les 15s)──> /actuator/prometheus

    L'application n'envoie rien • Prometheus vient chercher. Avantages :

    • contrôle centralisé de la fréquence
    • l'application ne dépend pas de Prometheus
    • résilience si Prometheus est down

    Format des métriques exposées

    # HELP booking_created_total Nombre de bookings créés
# TYPE booking_created_total counter
booking_created_total{application="kore-hexagonal",status="success"} 42.0
booking_created_total{application="kore-hexagonal",status="error"} 3.0

# HELP jvm_memory_used_bytes Used bytes of a given JVM memory area
jvm_memory_used_bytes{area="heap"} 1.24E8

    Les tags Micrometer deviennent des labels Prometheus • ils permettent de filtrer et agréger dans Grafana.

    Grafana • visualisation et alertes

    Grafana interroge Prometheus via PromQL et affiche les données en dashboards.

    Exemples de requêtes PromQL

    # Taux de bookings créés par seconde
rate(booking_created_total{status="success"}[5m])

# Taux d'erreur
rate(booking_created_total{status="error"}[5m])
  / rate(booking_created_total[5m])

# Mémoire heap utilisée
jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} * 100

    Métriques JVM exposées automatiquement

    Spring Boot + Micrometer expose sans configuration supplémentaire :

    • jvm_memory_used_bytes • mémoire heap et non-heap
    • jvm_gc_pause_seconds • durée des garbage collections
    • jvm_threads_live_threads • threads actifs
    • process_cpu_usage • usage CPU
    • http_server_requests_seconds • latence des endpoints HTTP

    Production mindset

    Les métriques permettent de répondre à une question que les logs ne peuvent pas :

    Le système est-il en bonne santé en ce moment ?

    Les logs disent ce qui s'est passé. Les métriques disent ce qui se passe.

    Alertes à configurer en priorité :

    Métrique Seuil d'alerte Signification
    Taux d'erreur HTTP > 5% Dégradation du service
    Mémoire heap > 85% Risque d'OutOfMemoryError
    Latence P99 > 2s Dégradation de l'expérience
    Consumer Kafka lag > 1000 Consumer en retard
    Health check DOWN Dépendance indisponible

    Conclusion

    La chaîne Actuator → Micrometer → Prometheus → Grafana est le standard de l'observabilité Spring Boot en production.

    Deux niveaux de health checks coexistent :

    • Actuator /actuator/health : pour les applications web, vérifié par les load balancers et orchestrateurs (Kubernetes)
    • BatchHealthAggregator (kore-batch) : pour les batchs, vérifié avant chaque lancement pour éviter de démarrer dans un environnement dégradé
