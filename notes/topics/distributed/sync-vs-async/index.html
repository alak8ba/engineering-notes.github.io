    Synchrone vs Asynchrone dans les Architectures Backend

    Note de contexte (12 mai 2026) : cette note constitue un retour d'expérience suite à des échanges avec des experts du domaine. Elle est rédigée le jour même et vise à structurer les notions abordées pour consolidation et montée en compétence.

    La question fondamentale

    Quand deux composants doivent communiquer, un seul choix structurant se pose :

    est-ce que l'appelant attend la réponse, ou continue-t-il sans attendre ?

    Ce choix impacte la performance, la résilience, la scalabilité et la complexité du système.

    Communication synchrone

    L'appelant bloque jusqu'à recevoir la réponse.

    Service A ──request──> Service B
Service A  <──response── Service B
    |
    (reprend l'exécution)

    Exemples

    • REST HTTP (GET /bookings/123 → attend le JSON)
    • gRPC (appel RPC bloquant)
    • Appel de méthode Java classique

    Implémentation dans kore-hexagonal • write side

    @RestController
@RequestMapping("/api/v1/bookings")
public class BookingController {

    @PostMapping
    public ResponseEntity<BookingView> create(
            @Valid @RequestBody CreateBookingRequest request) {

        UUID id = createHandler.handle(new CreateBookingCommand(
                request.assetRef(), request.startDate(), request.endDate()
        ));

        // Le client attend la réponse • flux synchrone
        return ResponseEntity.status(201).body(new BookingView(id));
    }
}

    Le handler est @Transactional • il persiste le booking et retourne l'UUID. Le client reçoit une réponse HTTP 201 avec l'identifiant créé.

    Avantages

    • simplicité conceptuelle et de debug
    • cohérence immédiate (le client sait si l'opération a réussi)
    • flux de contrôle clair

    Inconvénients

    • couplage temporel : si Service B est lent, Service A attend
    • propagation des pannes : si Service B est down, Service A échoue
    • latence cumulée dans les chaînes d'appels

    Communication asynchrone

    L'appelant publie un événement et continue sans attendre de réponse.

    Service A ──event──> [Broker Kafka]
Service A  (continue immédiatement)
                         |
                    Service B consomme
                    (plus tard, à son rythme)

    Implémentation dans kore-hexagonal • publication d'événement

    @Service
@RequiredArgsConstructor
public class CreateBookingHandler {

    private final BookingRepository bookingRepository;
    private final EventProducer eventProducer;

    @Transactional
    public UUID handle(CreateBookingCommand cmd) {
        // 1. Logique métier synchrone
        Booking booking = Booking.create(
            cmd.assetRef(), cmd.startDate(), cmd.endDate());
        bookingRepository.save(booking);

        // 2. Publication asynchrone des événements de domaine
        booking.pullEvents().forEach(eventProducer::publish);

        return booking.getId();
        // Le handler ne sait pas ce que les consumers vont faire
    }
}

    @Override
public void publish(DomainEvent event) {
    EventEnvelope envelope = new EventEnvelope(
            UUID.randomUUID(),
            event.eventType(),
            ZonedDateTime.now(),
            objectMapper.valueToTree(event)
    );
    // Publication dans Kafka • le producer n'attend pas la consommation
    kafkaTemplate.send(topic, envelope.eventId().toString(),
                       objectMapper.writeValueAsString(envelope));
}

    Le consumer (BookingEventHandler) met à jour le read model de façon indépendante, sans bloquer le write side.

    Avantages

    • découplage temporel : le producer ne dépend pas de la disponibilité des consumers
    • résilience : si un consumer est down, les messages s'accumulent et sont traités au redémarrage
    • scalabilité : on ajoute des consumers sans toucher au producer

    Inconvénients

    • cohérence éventuelle uniquement
    • debugging plus difficile (flux non linéaire)
    • besoin d'observabilité forte (correlationId, tracing)

    Pattern hybride • le cas réel de kore-hexagonal

    En production, les systèmes n'utilisent pas l'un ou l'autre exclusivement. kore-hexagonal est un exemple typique de pattern hybride :

    [Client HTTP]
      |
  REST sync       ← création du booking, réponse 201 immédiate
      |
[CreateBookingHandler]
      |
  JPA save        ← écriture synchrone en base
      |
  Kafka publish   ← déclenchement asynchrone du read model
      |
[BookingEventConsumer]  ← traitement asynchrone, indépendant
      |
  booking_read_model mise à jour

    Lecture (GET) : synchrone via BookingReadController → lit directement booking_read_model

    Ecriture (POST/PUT) : synchrone pour la persistance métier, asynchrone pour la propagation aux consumers

    Ce choix est délibéré : - le client a une réponse immédiate sur le résultat de son action - la mise à jour du read model ne bloque pas l'écriture - si le consumer est lent, l'écriture n'en pâtit pas

    Gestion des erreurs • comportement différent

    Synchrone

    L'erreur remonte directement dans la stack d'appel :

    POST /bookings → BookingDomainException → GlobalExceptionHandler → HTTP 400

    Le client sait immédiatement que son action a échoué.

    Asynchrone

    L'erreur ne peut pas remonter vers le client • il est déjà reparti :

    Kafka message → consumer plante → retry (x3) → DLT → log + alerte Grafana

    Le client ne saura jamais que le read model n'a pas été mis à jour, sauf si on lui envoie une notification.

    Cohérence des données

    Modèle Cohérence Performance Cas d'usage
    Synchrone Forte (immédiate) Plus lente Opérations critiques, validation
    Asynchrone Eventuelle Plus rapide Notifications, projections, pipelines

    Dans kore-hexagonal, le write model (base principale) est fortement cohérent via @Transactional. Le read model est éventuellement cohérent • il peut avoir quelques millisecondes de retard sur le write model.

    Quand choisir quoi

    Situation Synchrone Asynchrone
    Le client a besoin du résultat immédiatement Oui Non
    L'opération peut être longue Non Oui
    Plusieurs services doivent réagir au même événement Non Oui
    La cohérence immédiate est critique Oui Non
    Le système doit absorber des pics de charge Non Oui

    Observabilité • plus critique en asynchrone

    En synchrone, la stack trace suffit pour débugger. En asynchrone, il faut :

    • un correlationId propagé de l'HTTP jusqu'aux events Kafka
    • des logs structurés avec ce correlationId dans chaque consumer
    • des métriques sur le consumer lag pour détecter les retards
    • un dashboard Grafana montrant le flux de bout en bout

    Sans cette observabilité, le debugging d'un système async devient extrêmement difficile.

    Conclusion

    Synchrone et asynchrone ne sont pas opposés • ils se complètent.

    • Synchrone pour les opérations où le client a besoin d'un résultat immédiat et fiable
    • Asynchrone pour propager les effets de bord, découpler les services, absorber la charge

    kore-hexagonal illustre ce pattern : écriture synchrone + propagation asynchrone via Kafka. C'est l'architecture standard des systèmes distribués modernes.
