    Fetch Strategies, Jointures & Read Models Event-Driven

    Note de contexte (12 mai 2026) : cette note constitue un retour d’expérience suite à des échanges avec des experts du domaine. Elle est rédigée le jour même et vise à structurer les notions abordées pour consolidation et montée en compétence.

    Contexte

    Cette note explore les différentes stratégies de chargement des données dans les applications backend modernes, notamment dans les contextes fortement orientés lecture et dashboards.

    Les échanges ont porté sur plusieurs problématiques liées à JPA / Hibernate et aux architectures distribuées :

    • stratégies de fetch (EAGER vs LAZY)
    • coût des jointures SQL
    • problématiques de performance et de scalabilité
    • surcharge liée aux vues enrichies côté frontend
    • optimisation des lectures
    • pré-calcul de projections
    • architectures event-driven et read models

    L’objectif est de comparer les approches ORM classiques basées sur les jointures dynamiques avec une approche orientée projections de lecture pré-calculées via événements Kafka et read models dédiés.

    Le problème fondamental

    Dans les applications backend, les données sont souvent fortement relationnelles.

    Exemple :

    • User
    • Order
    • Payment
    • Product
    • Reservation

    Les interfaces frontend ont généralement besoin d’afficher des vues enrichies contenant plusieurs relations.

    Cela crée rapidement des problématiques de :

    • jointures SQL coûteuses
    • surchargement mémoire
    • latence élevée
    • multiplication des requêtes
    • problèmes N+1

    Fetch Lazy

    Le fetch lazy charge les relations uniquement lorsqu’elles sont réellement demandées.

    Exemple conceptuel :

    @OneToMany(fetch = FetchType.LAZY)
private List<Order> orders;

    Fonctionnement

    • l’entité principale est chargée
    • les relations restent non chargées
    • Hibernate charge les données plus tard si nécessaire

    Avantages

    • réduction du volume initial chargé
    • meilleure utilisation mémoire
    • requêtes initiales plus légères

    Inconvénients

    • risque de N+1 queries
    • multiplication des accès base
    • LazyInitializationException possible
    • comportement parfois difficile à maîtriser

    Fetch Eager

    Le fetch eager charge immédiatement les relations.

    Exemple :

    @OneToMany(fetch = FetchType.EAGER)
private List<Order> orders;

    Fonctionnement

    • Hibernate récupère immédiatement les relations
    • souvent via des jointures SQL

    Avantages

    • données immédiatement disponibles
    • comportement plus simple à comprendre

    Inconvénients

    • surcharge mémoire
    • jointures SQL lourdes
    • sur-fetching
    • dégradation des performances
    • amplification du coût sur les listes / scrolls / dashboards

    Dans les applications fortement orientées lecture, le eager fetching peut rapidement devenir problématique.

    Le vrai problème : les vues frontend riches

    Les interfaces modernes demandent souvent :

    • dashboards
    • listes enrichies
    • vues agrégées
    • données consolidées

    Le backend finit alors par reconstruire constamment les mêmes jointures complexes.

    Cela entraîne :

    • coût CPU
    • coût base de données
    • augmentation de la latence
    • surcharge ORM

    Limites du modèle relationnel temps réel

    Dans une architecture classique :

    1. le frontend demande une vue
    2. le backend reconbine plusieurs tables
    3. Hibernate reconstruit les relations
    4. les jointures sont recalculées à chaque requête

    Cette approche fonctionne bien au début, mais devient coûteuse à grande échelle.

    Approche alternative : Read Models pré-calculés

    Une autre approche consiste à préparer les vues de lecture à l’avance.

    Le principe :

    • les événements métier sont publiés via Kafka
    • un consumer construit un read model optimisé
    • le frontend lit directement ce modèle

    Architecture Event-Driven

    Exemple simplifié :

    1. ReservationCreated
    2. ReservationUpdated
    3. PaymentConfirmed
    4. événements publiés dans Kafka
    5. consumer met à jour un read model dédié

    Le read model contient déjà :

    • les données agrégées
    • les relations nécessaires
    • les champs optimisés pour l’UI

    Objectif du Read Model

    Le read model sert à :

    • éviter les jointures répétitives
    • réduire les fetch complexes
    • optimiser les lectures frontend
    • accélérer les dashboards
    • réduire la charge ORM

    Le frontend récupère directement une structure prête à afficher.

    Comparaison des approches

    Fetch dynamique (lazy/eager)

    Avantages

    • simple à mettre en place
    • cohérence immédiate
    • modèle relationnel classique

    Inconvénients

    • coût runtime élevé
    • jointures répétées
    • problèmes de performance à grande échelle
    • dépendance forte à l’ORM

    Read Models Event-Driven

    Avantages

    • lectures rapides
    • données pré-agrégées
    • réduction du coût des jointures
    • meilleure scalabilité lecture
    • optimisation des dashboards

    Inconvénients

    • complexité architecturale plus élevée
    • cohérence éventuelle
    • duplication contrôlée des données
    • besoin de pipeline événementiel fiable

    CQRS léger

    Cette approche se rapproche d’un modèle CQRS :

    • write model → logique métier
    • read model → optimisation lecture

    Le read model devient une projection adaptée aux besoins du frontend.

    Implémentation réelle • kore-hexagonal

    Ce pattern est implémenté dans kore-hexagonal sur le domaine Booking.

    Flux complet

    [Write side]                          [Read side]
CreateBookingCommand                  GET /api/v1/projection/bookings
        |                                        |
CreateBookingHandler                  BookingReadController
        |                                        |
Booking (domain)                      BookingReadJpaRepository
        |                                        |
BookingCreatedEvent                   booking_read_model (table)
        |                                        ^
KafkaEventProducer                               |
        |                             BookingEventHandler
        +---------> Kafka topic ------+

    1. L’événement de domaine

    public record BookingCreatedEvent(
        UUID bookingId,
        String assetRef,
        LocalDate startDate,
        LocalDate endDate,
        BookingStatus status
) implements AggregateEvent {}

    L’événement est produit par le write side après persistance du Booking.

    2. La table de lecture dénormalisée

    @Entity
@Table(name = "booking_read_model")
public class BookingReadJpaEntity {

    @Id
    private UUID id;

    private String assetRef;
    private LocalDate startDate;
    private LocalDate endDate;

    @Enumerated(EnumType.STRING)
    private BookingStatus status;

    private ZonedDateTime lastUpdatedAt;

    public void update(String assetRef, LocalDate startDate,
                       LocalDate endDate, BookingStatus status) {
        this.assetRef = assetRef;
        this.startDate = startDate;
        this.endDate = endDate;
        this.status = status;
        this.lastUpdatedAt = ZonedDateTime.now();
    }
}

    Pas de jointures. La table contient exactement ce dont le frontend a besoin.

    3. Le consumer Kafka qui maintient le read model

    @Service
public class BookingEventHandler {

    private final BookingReadJpaRepository readModelRepository;
    private final ProcessedEventJpaRepository processedEventRepository;

    public void handle(EventEnvelope envelope) {

        // Idempotence : ignorer les doublons
        if (processedEventRepository.existsByEventId(envelope.eventId())) {
            log.warn("Evenement duplique ignore : {}", envelope.eventId());
            return;
        }

        switch (envelope.eventType()) {

            case "BookingCreatedEvent" -> {
                BookingCreatedEvent e = mapper.convertValue(
                    envelope.payload(), BookingCreatedEvent.class);
                readModelRepository.save(new BookingReadJpaEntity(
                    e.bookingId(), e.assetRef(), e.startDate(),
                    e.endDate(), e.status()
                ));
            }

            case "BookingUpdatedEvent" -> {
                BookingUpdatedEvent e = mapper.convertValue(
                    envelope.payload(), BookingUpdatedEvent.class);
                BookingReadJpaEntity entity = readModelRepository
                    .findById(e.bookingId())
                    .orElseGet(() -> new BookingReadJpaEntity(...));
                entity.update(e.assetRef(), e.startDate(),
                              e.endDate(), e.status());
                readModelRepository.save(entity);
            }
        }
    }
}

    Points clés :

    • L’idempotence est gérée via une table processed_events : même si Kafka redélivre l’événement, le read model n’est mis à jour qu’une seule fois
    • Le switch sur eventType rend le handler extensible sans modifier la logique existante
    • Le read model est mis à jour de façon asynchrone, sans bloquer le write side

    4. Le controller de lecture • séparé du write

    /**
 * Endpoints lecture seule • Read model CQRS.
 * Retourne des projections depuis booking_read_model,
 * sans passer par le domaine d’écriture.
 */
@RestController
@RequestMapping("/api/v1/projection/bookings")
public class BookingReadController {

    @GetMapping("/{id}")
    public ResponseEntity<BookingReadView> getById(@PathVariable UUID id) {
        return ResponseEntity.ok(
            mapper.toReadView(getHandler.handle(new GetBookingQuery(id)))
        );
    }

    @GetMapping
    public ResponseEntity<BookingPageView> getPage(
            @RequestParam(defaultValue = "0") int page,
            @RequestParam(defaultValue = "20") int size,
            BookingFilterRequest filter) {
        // Lecture directe depuis booking_read_model, sans jointure
        return ResponseEntity.ok(mapper.toPageView(getHandler.handle(query)));
    }
}

    Le prefix /projection/ signale explicitement qu’on lit depuis le read model, pas depuis le domaine.

    Ce que ça évite concrètement

    Sans read model, chaque appel à GET /bookings forcerait :

    SELECT b.*, bi.*
FROM booking b
LEFT JOIN booking_item bi ON bi.booking_id = b.id
WHERE b.status = ‘CONFIRMED’
ORDER BY b.start_date

    Avec jointure recalculée à chaque requête, à chaque scroll, à chaque refresh dashboard.

    Avec le read model :

    SELECT * FROM booking_read_model
WHERE status = ‘CONFIRMED’
ORDER BY start_date

    Une seule table, aucune jointure, index direct sur les colonnes de filtre.

    Trade-off important

    Le choix n’est pas :

    eager vs lazy uniquement

    Le vrai sujet est :

    où veut-on payer le coût de recomposition des données ?

    Approche Quand payer le coût Cohérence
    Fetch dynamique (lazy/eager) A chaque requête Forte (immédiate)
    Read model event-driven A la production de l’événement Eventuelle

    Production mindset

    Dans les systèmes à forte volumétrie (dashboards, scroll infini, monitoring temps réel) :

    • pré-calculer les vues via événements est plus efficace que reconstruire les relations ORM à chaque requête
    • la cohérence éventuelle est acceptable quand les données de lecture n’ont pas besoin d’être à la milliseconde près
    • l’idempotence du consumer est non négociable en production • Kafka peut redélivrer

    Conclusion

    Le fetch lazy et eager répondent à des problématiques ORM classiques, mais montrent rapidement leurs limites dans les architectures distribuées modernes.

    Les architectures event-driven avec read models pré-calculés permettent de déplacer le coût de recomposition des données en amont afin d’optimiser les lectures et améliorer la scalabilité.

    Cette approche introduit davantage de complexité architecturale, mais devient particulièrement pertinente pour les systèmes orientés dashboards, analytics et forte charge de lecture.
