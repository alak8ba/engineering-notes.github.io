    Stabilité en production

    Ce document est une version adaptée d'une note de développement réelle. Certains détails techniques ont été nuancés ou généralisés à des fins de publication.

    Contexte

    Amélioration de la stabilité et de la robustesse de la plateforme en production.

    Travaux réalisés sur l'infrastructure Docker, la configuration Nginx, la gestion des erreurs frontend et l'optimisation de la pagination.

    Infrastructure

    Stack Docker

    Vérification et stabilisation de la stack complète en production :

    • PostgreSQL
    • Kafka / Zookeeper
    • Backend Spring Boot
    • Frontend Nginx

    Utilisation de Watchtower pour les mises à jour automatiques des images • le pipeline CI/CD publie une nouvelle image, Watchtower la déploie automatiquement.

    Nginx • Ajustement du rate limiting

    Les zones de rate limiting ont été recalibrées pour trouver l'équilibre entre :

    • sécurité • limiter les abus et les attaques
    • performance • éviter les faux positifs 429 Too Many Requests sur un usage normal

    Les endpoints de pagination nécessitent un seuil plus élevé car le frontend effectue plusieurs requêtes en séquence rapide lors du scroll.

    Frontend • Pagination optimisée

    Refactoring du composant de liste principal.

    Améliorations :

    • mise en cache des pages déjà chargées
    • prefetch de la page suivante en arrière-plan
    • prévention des requêtes doublons sur la même page
    • gestion des états de chargement par page

    Résultat : navigation fluide sans blocages ni requêtes redondantes.

    Gestion des erreurs API

    Création d'une classe de normalisation des erreurs API.

    Objectif : transformer les codes d'erreur HTTP en messages compréhensibles pour l'utilisateur.

    Exemple :

    Code HTTP Message affiché
    429 Trop de requêtes, veuillez patienter
    401 Session expirée, reconnexion requise
    500 Erreur serveur, réessayez plus tard

    Intercepteur Axios • Refresh automatique

    Implémentation d'un intercepteur Axios pour gérer automatiquement le cycle de vie des tokens :

    1. requête API échoue avec 401
    2. intercepteur tente un refresh token silencieux
    3. si succès → retry automatique de la requête initiale
    4. si échec → redirection vers le login

    Les endpoints d'authentification sont exclus de ce mécanisme pour éviter les boucles infinies.

    CI/CD

    La configuration Nginx est versionnée directement dans le projet frontend.

    Pipeline de déploiement :

    build image → push registry → Watchtower détecte → déploiement automatique

    Avantages :

    • configuration reproductible et versionnée
    • déploiement sans intervention manuelle
    • rollback possible via Git

    Résultat

    Plateforme plus robuste en production :

    • pagination rapide sans blocages
    • erreurs API claires côté utilisateur
    • rate limiting calibré sans faux positifs
    • déploiement continu automatisé
